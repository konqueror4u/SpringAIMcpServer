## McpServerApplication

**File**: `src/main/java/org/vis/mcpserver/McpServerApplication.java`

### 1. main()

```
Method Name: main
Parameters: String[] args - Command line arguments passed to the application
Return Type: void
Annotations: -
```

**Usage**: Serves as the entry point for the Spring Boot application. This method bootstraps the application context, initializes all Spring beans, starts the embedded server, and loads all configurations from `application.yml`.

**Execution Flow**:
1. Starts the Spring Boot application context
2. Initializes all Spring beans and dependencies
3. Loads `application.yml` configuration
4. Starts the MCP server on port 9060
5. Initializes database connection pool
6. Registers AI tools via Tool Callback Provider

---

### 2. tools()

```
Method Name: tools
Parameters: 
  - SellerAccountTools sellerAccountTools (The service containing all AI tool definitions)
Return Type: ToolCallbackProvider
Annotations: @Bean
```

**Usage**: Creates a Spring Bean that registers and manages all seller account tools. This provider exposes the seller account operations (search by name, search by owner, fetch all) as callable AI tools that can be invoked via the MCP protocol. The `MethodToolCallbackProvider` scans the `SellerAccountTools` service and registers all methods annotated with `@Tool` annotation.

**Bean Lifecycle**:
- Instantiated during Spring context initialization
- Scans `SellerAccountTools` for `@Tool` annotated methods
- Registers 3 AI tools: `search_seller_accounts_by_name`, `fetch_all_seller_accouts`, `search_seller_accounts_by_owner`
- Made available for MCP clients to invoke

---

### 3. mcpAsyncServer()

```
Method Name: mcpAsyncServer
Parameters:
  - McpStreamableServerTransportProvider transport (The HTTP transport provider for streaming MCP messages)
Return Type: McpAsyncServer
Annotations: @Bean
```

**Usage**: Creates and configures the Model Context Protocol (MCP) async server bean. This server handles all incoming MCP requests via the Streamable HTTP protocol. The `transport` parameter is automatically provided by Spring AI framework as a streamable HTTP implementation that listens on configured endpoints (`/mcp`, `/mcp/message`, `/sse`). This enables bidirectional communication with MCP clients for tool invocation.

**Server Configuration**:
- Protocol: STREAMABLE (HTTP-based)
- Async Type: ASYNC (non-blocking)
- Endpoints:
  - `/mcp` - Main MCP endpoint
  - `/mcp/message` - MCP message endpoint (POST)
  - `/sse` - Server-Sent Events endpoint

## DatabaseConfig

**File**: `src/main/java/org/vis/mcpserver/config/DatabaseConfig.java`

### 1. initializer()

```
Method Name: initializer
Parameters:
  - ConnectionFactory connectionFactory (The R2DBC connection factory for PostgreSQL)
Return Type: ConnectionFactoryInitializer
Annotations: @Bean, @Configuration
```

**Usage**: Configures and creates a `ConnectionFactoryInitializer` bean that automatically executes the `schema.sql` file when the application starts. This initializer:
- Connects to the PostgreSQL database using the provided ConnectionFactory
- Populates the database with the schema from `schema.sql` (creates tables, indexes, etc.)
- Runs only once during application startup for database setup

**Initialization Process**:
1. Creates a new `ConnectionFactoryInitializer` instance
2. Sets the R2DBC ConnectionFactory (postgres://localhost:5433)
3. Creates a `ResourceDatabasePopulator` with `schema.sql`
4. Executes SQL scripts on application startup
5. Creates the `seller_account` table
6. Inserts sample data (5 test sellers)

**SQL Operations**:
```sql
CREATE SCHEMA IF NOT EXISTS postgres
CREATE TABLE IF NOT EXISTS postgres.seller_account (...)
INSERT INTO postgres.seller_account VALUES (...)
```

- Nullable: No

## SellerAccountRepository

**File**: `src/main/java/org/vis/mcpserver/repository/SellerAccountRepository.java`

**Parent Interface**: `ReactiveCrudRepository<SellerAccount, Long>`

This is a reactive repository interface that extends Spring Data R2DBC's ReactiveCrudRepository, providing non-blocking database operations.

### 1. findByName()

```
Method Name: findByName
Parameters:
  - String name (The seller account name to search for - exact match)
Return Type: Flux<List<SellerAccount>>
Annotations: -
```

**Usage**: Performs an asynchronous, non-blocking database query to find all seller accounts that match the given name. Returns a `Flux` (reactive stream) that emits a list of matching `SellerAccount` objects. This method uses Spring Data R2DBC reactive repository pattern for non-blocking database operations.

**Query Behavior**:
- Searches for exact name matches
- Returns Flux (reactive stream)
- Non-blocking execution
- Can be subscribed to for async processing

**Example Usage**:
```java
findByName("harry")           // Returns all accounts named "harry"
findByName("mike")            // Returns all accounts named "mike"
```

**Generated SQL** (approximate):
```sql
SELECT * FROM seller_account WHERE name = ?
```

---

### 2. findByOwner()

```
Method Name: findByOwner
Parameters:
  - String owner (The owner/organization name to search for)
Return Type: Flux<List<SellerAccount>>
Annotations: -
```

**Usage**: Performs an asynchronous, non-blocking database query to find all seller accounts that belong to a specific owner/organization. Returns a `Flux` (reactive stream) that emits a list of all `SellerAccount` objects owned by the specified owner. This is useful for querying all sellers managed by a particular marketplace or organization.

**Query Behavior**:
- Searches for owner field matches
- Returns Flux (reactive stream)
- Non-blocking execution
- Efficient resource utilization in high-concurrency environments

**Example Usage**:
```java
findByOwner("amazon")         // Returns all Amazon-managed accounts
findByOwner("flipkart")       // Returns all Flipkart-managed accounts
findByOwner("manomano")       // Returns all ManoMano-managed accounts
```

**Generated SQL** (approximate):
```sql
SELECT * FROM seller_account WHERE owner = ?
```

## SellerAccountTools

---

### 2. getAccountByName()

```
Method Name: getAccountByName
Parameters:
  - String name (The seller account name to search for - Tool Parameter)
Return Type: String (JSON string representation)
Annotations: @Tool(name = "search_seller_accounts_by_name", description = "Find all Seller Accounts by name")
              @ToolParam(description = "Seller Account Name")
```

**Tool Definition**:
- **Tool Name**: `search_seller_accounts_by_name`
- **Tool Description**: Find all Seller Accounts by name
- **Parameter Name**: name
- **Return Format**: JSON

**Usage**: This is an AI-callable tool exposed via the MCP protocol. When an AI/LLM client invokes the `search_seller_accounts_by_name` tool with a name parameter, this method executes a reactive query to find matching accounts. The results are converted to JSON format for the AI to process. Used for intelligent seller lookups based on business name. Returns JSON array of matching SellerAccount objects.

**Execution Flow**:
1. Receives seller account name from AI client
2. Calls `sellerAccountRepository.findByName(name)`
3. Subscribes to reactive stream
4. Collects results into List
5. Blocks to wait for completion (converts Flux to List)
6. Serializes List to JSON string using ObjectMapper
7. Returns JSON to AI client

**Example Invocation**:
```
Tool: search_seller_accounts_by_name
Parameter: {"name": "harry"}
Returns: [{"id": 1, "name": "harry", "owner": "manomano", ...}]
```

---

### 3. getAllAccount()

```
Method Name: getAllAccount
Parameters: None (This tool takes NO parameters)
Return Type: List<SellerAccount>
Annotations: @Tool(name = "fetch_all_seller_accouts", description = "Fetches ALL seller accounts from the database. This tool takes NO parameters")
```

**Tool Definition**:
- **Tool Name**: `fetch_all_seller_accouts`
- **Tool Description**: Fetches ALL seller accounts from the database. This tool takes NO parameters
- **Parameter Count**: 0
- **Return Format**: List<SellerAccount>

**Usage**: This is an AI-callable tool that retrieves all seller accounts from the database. The results are limited to the first 5 records using `.take(5)` to prevent overwhelming responses. This tool is useful when an AI needs a general overview of available sellers without filtering by specific criteria. The reactive chain blocks at the end to convert the asynchronous Flux stream to a synchronous List for the tool response.

**Execution Flow**:
1. Calls `sellerAccountRepository.findAll()` - returns Flux of ALL records
2. Applies `.take(5)` - limits stream to first 5 items
3. Calls `.collectList()` - collects into List
4. Calls `.block()` - blocks to wait for completion
5. Returns List<SellerAccount> with up to 5 records

**Limitations**:
- Limited to 5 records for performance and response size optimization
- Prevents overwhelming AI responses
- First 5 records from insertion order

**Example Invocation**:
```
Tool: fetch_all_seller_accouts
Parameters: (none)
Returns: List of 5 SellerAccount objects
```

## 📊 Method Invocation Flow Diagram

```
MCP Client Request
    ↓
MCPAsyncServer (handles HTTP)
    ↓
ToolCallbackProvider (routes to correct tool)
    ↓
SellerAccountTools (executes AI tool method)
    ↓
SellerAccountRepository (reactive database query)
    ↓
PostgreSQL Database
    ↓
Result → JSON → ToolCallbackProvider → MCPAsyncServer → MCP Client Response
```

## 📝 Notes

- All database operations are **reactive/non-blocking** using R2DBC and Reactor framework
- All AI tools return JSON-serialized responses for AI/LLM processing
- Database schema is automatically initialized on application startup
- The project uses Spring Boot 4.0.5 and Spring AI 2.0.0-M4
- Java 26 is required (LTS version)
